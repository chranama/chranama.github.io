System Overview

llm-extraction-platform is a production-style LLM application service that exposes generate and schema-constrained extract capabilities with explicit contracts, policy-aware runtime behavior, durable async jobs, trace inspection, and a proof-oriented evidence system.

The flagship proof inside that broader system is extract capability gating: offline onboarding artifacts determine whether extraction is enabled at runtime, and the result is verified through deterministic PASS/FAIL evidence artifacts. But the case study is bigger than that single feature. It also shows Kubernetes deployability, separate-worker async execution, and traceable request inspection across API and worker boundaries.

This backend is also the application side of a larger runtime story. It is intentionally designed to remain a valid standalone inference service even when fronted by inference-serving-gateway, which owns edge/runtime concerns such as admission, timeouts, readiness, and gateway-side observability.

Why This System Matters

LLM APIs need more than a model call wrapped in an endpoint. To be credible as backend or platform work, they need explicit contracts, runtime controls, operational inspection, and a way to prove that important behaviors are real.

This system is strongest as a backend/platform case study because it ties together product-facing API behavior, offline-to-runtime control signals, deployment and execution modes, and proof artifacts that reduce ambiguity for both operators and reviewers.

Lead Proof: Extraction Capability Gating

The lead proof surface shows how offline onboarding and evaluation artifacts set extraction capability in patched model manifests, and how runtime policy checks that capability before serving extraction requests.

The canonical signal is deterministic PASS/FAIL evidence output rather than a vague claim about model readiness.

  • Run python proof/validate_evidence_manifest.py.
  • Inspect proof/evidence_manifest.latest.json to confirm that the PASS runtime is extract-capable while the FAIL runtime is capability-blocked.

Supporting System Proofs

Kubernetes deployment proof. The same service has a canonical Kubernetes reviewer path. This proof shows that the system is not limited to local-only execution: the local kind deployment passes smoke checks and the production overlay renders cleanly, which makes the deployment story much more credible than a demo-only setup.

Async execution proof. The same system supports queued extraction through a separate worker process. This proof matters because it shows the system can separate request submission from background execution while still exposing durable job state; the canonical signal is 202 submission, separate worker execution, and a final succeeded job result.

Trace inspection proof. The same system supports request-level inspection for both sync and async extraction. This proof gives the system a strong observability story: a single trace id links submit, queue, worker claim, execution, and final status across the API-to-worker boundary, so behavior stays inspectable even when work crosses processes.

5-10 Minute Reviewer Path

  1. Open the reviewer path.
  2. Run python proof/generate_canonical_manifest.py.
  3. Inspect PASS/FAIL gating artifacts in proof/evidence_manifest.latest.json.
  4. Inspect the Kubernetes proof in proof/artifacts/phase5_k8s_kind/.
  5. Inspect the async extract proof in proof/artifacts/phase6_extract_async/.
  6. Inspect the request-trace proof in proof/artifacts/phase7_trace_inspection/.
