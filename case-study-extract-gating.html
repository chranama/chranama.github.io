Case Study

Extract Capability Gating

LLM Extraction Platform

This case study shows how model capability onboarding and runtime policy enforcement prevent unsupported extraction behavior and produce reviewer-friendly evidence artifacts.

Back to Portfolio

Problem

Extraction endpoints can appear to work even when a model is not properly onboarded for schema-constrained output. That creates silent reliability risk: invalid structures, brittle outputs, and unclear operator confidence.

The platform treats this as a policy and capability problem, not just a prompt-engineering problem.

Approach

  • Offline evaluation/onboarding artifacts set model capability in patched model manifests.
  • Runtime policy checks capability before serving extraction requests.
  • Demo scripts execute PASS/FAIL flows and write deterministic evidence manifests.

Project demos | Policy docs | Server docs

Architecture and Flow

  1. Run extract-gate demo script: scripts/demo_extract_gate/run_host_transformers.sh.
  2. Script executes a host deployment scenario with supported and unsupported model capability paths.
  3. Server enforcement allows extraction only when onboarding capability is present.
  4. Artifacts are written to traffic_out/<run>/ including PASS/FAIL runtime outputs and manifest.

Verification Steps

How a reviewer verifies this in under 10 minutes

  1. Open the reviewer path.
  2. Run Demo 2: Extract Gate.
  3. Inspect these outputs:
    • `traffic_out/<run>/evidence_manifest.json` (path reference)
    • `traffic_out/<run>/host_pass_runtime.json` (review flow)
    • `traffic_out/<run>/host_fail_runtime.json` (review flow)
