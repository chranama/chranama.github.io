LLM Extraction Platform

  1. Context: extraction can appear to work on unsupported models.
    Lesson: prompt-level behavior is not enough for reliability guarantees.
    Decision: enforce capability at runtime through policy gating.
    Evidence: proof/evidence_manifest.latest.json, PASS/FAIL runtime outputs in traffic_out/....
  2. Context: PASS and FAIL paths can fail for different reasons (policy block vs runtime/model load issues).
    Lesson: failure mode classification matters as much as pass/fail status.
    Decision: preserve runtime probes and manifests so capability guardrail failures are distinguishable from runtime failures.
    Evidence: model decision memo + host_pass_extract.json/host_fail_extract.json artifacts.
  3. Context: multi-runtime, multi-lane CI can create confidence gaps if checks are weak or non-reproducible.
    Lesson: portfolio credibility depends on strict quality lanes and local reproducibility.
    Decision: enforce no-noop core checks, compact failure artifact bundles, and deterministic local CI smoke matrix.
    Evidence: CI hardening doc + scripts/ci_smoke_matrix.sh + required core test lanes.

BioLLM-Finetune (Thesis-Derived)

  1. Context: thesis BioASQ results were uneven by question type (yes/no strongest, factoid/list weakest).
    Lesson: one aggregate score hides the real failure modes of generative biomedical QA.
    Decision: evaluate by type and metric, then add explicit robustness slices (phenotype and perturbation views).
    Evidence: thesis Results chapter reports yes/no peak accuracy of 64.0% (Macro-F1 0.5486), factoid strict accuracy up to 15.8%, and near-zero list scores.
  2. Context: formatting and schema mismatches caused large evaluation penalties even when answers were semantically close.
    Lesson: output contracts and post-processing are core system components, not cosmetic cleanup.
    Decision: enforce structured output normalization and retain deterministic evidence artifacts for verification.
    Evidence: thesis sections on error analysis + post-processing impact, plus canonical artifacts in proof/evidence_manifest.latest.json.
  3. Context: high-capability systems still had factual grounding limits in closed-book generation.
    Lesson: reliability requires retrieval/knowledge grounding and richer evaluation, not just stronger prompting.
    Decision: roadmap emphasizes RAG, ontology-informed validation, and human-in-the-loop quality assessment.
    Evidence: thesis Conclusion and Future Work chapter + current reproducibility pipeline in the repo.

California-Portugal Climate Pipeline

  1. Context: notebook-first ML work breaks down during repeated runs.
    Lesson: orchestration and layer contracts are non-optional for reliability.
    Decision: use dbt layered models + Prefect workflows + persisted run outputs.
    Evidence: src/climate_pipeline/orchestration/prefect_flow.py, dbt model layers, canonical manifest.
  2. Context: baseline model metrics are misleading when feature tables drift or null handling changes.
    Lesson: schema/feature integrity checks must gate model trust.
    Decision: test required feature columns, target binary constraints, and no-NaN guarantees before treating metrics as valid.
    Evidence: tests/test_ml_features.py and anomaly-layer schema tests.
  3. Context: pipeline freshness and model quality degrade silently without run-level observability.
    Lesson: operational metadata (deltas, freshness, run mode) is part of ML reliability proof.
    Decision: log pipeline run records and ML metrics to warehouse observability tables and assert them in tests.
    Evidence: src/climate_pipeline/observability/run_logger.py, tests/test_observability_models.py, tests/test_ml_training_smoke.py.
