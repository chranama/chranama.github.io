System Overview

inference-serving-gateway is a Go edge service that fronts inference backends and owns gateway concerns: request admission, timeout and cancellation handling, concurrency and rate controls, request and trace propagation, readiness checks, structured edge errors, and observability surfaces.

The project is intentionally scoped around the service boundary rather than model semantics. Extraction semantics, async job execution, and backend trace detail stay owned by llm-extraction-platform; the gateway owns edge/runtime behavior in front of that backend.

The companion system is the llm-extraction-platform backend case study, which shows the application-owned side of the same split: extraction semantics, async jobs, policy-aware runtime behavior, and traceable backend execution.

Why This System Matters

Many AI projects prove application behavior but not the serving boundary in front of the application. This system matters because real AI products often need an edge layer that can enforce limits, preserve request identity, reflect upstream readiness, and translate upstream failures into gateway-owned runtime behavior.

As a case study, it is strongest for backend and platform hiring because it shows practical distributed-systems judgment without overstating the scope: service boundaries, failure handling, HTTP contracts, and runtime controls are treated as first-class engineering concerns.

Lead Proof: Canonical Mock-Upstream Gateway Path

The canonical v1 proof path runs the gateway against a mock upstream and captures the full edge contract: /healthz, /readyz, /metrics, sync forwarding, async submit, async status polling, and request/trace identity continuity.

  • Run proof/generate_mock_proof.sh.
  • Inspect proof/artifacts/mock_upstream/latest/summary.md and manifest.json to confirm stable route behavior, readiness semantics, metrics exposure, and sync/async request identity continuity.

Supporting Proof: Live Backend Compatibility

The stronger follow-on proof is that the gateway contract is not only valid against a mock upstream, but also compatible with the real llm-extraction-platform backend. The integration probe shows that request and trace continuity, sync extract, async submit, and async status flows remain coherent across the split gateway/backend design.

This matters because it demonstrates that the gateway is not just a toy HTTP proxy. It preserves a real service boundary while staying compatible with a separate backend that owns extraction semantics and job execution.
